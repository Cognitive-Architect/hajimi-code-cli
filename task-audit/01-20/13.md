【特殊审计关注点】

1. Token Bucket算法正确性（高风险）
审计重点：
- `refillRate`是否严格为100/60（每秒1.666...个token）
- `consume`是否先补充再消费（而非直接减）
- `resetTime`计算是否准确（下次补充时间点）

验证命令：

```bash
node -e "
const {TokenBucketRateLimiter} = require('./src/security/rate-limiter');
const limiter = new TokenBucketRateLimiter({capacity: 20, refillRate: 100/60});
// 连续消费21次，验证第21次是否拒绝
for(let i=0; i<21; i++) {
  const r = limiter.consume('127.0.0.1');
  console.log(i+1, r.allowed, r.remaining);
}
"
```

2. 超时资源释放（中风险）
审计重点：
- 正常响应时`res.on('finish')`是否清理timer
- 客户端断开时`res.on('close')`是否清理timer
- 是否存在timer泄漏导致内存增长

验证方法：
代码审查 + 压测后检查`process._getActiveHandles()`中timer数量是否稳定

3. 敏感信息掩码完备性（高风险）
审计重点：
- 是否覆盖全部5类字段：password, secret, api_key, token, authorization
- 是否防御大小写变体（Password, PASSWORD）
- 是否防御嵌套对象（`{user: {password: 'x'}}`）

验证命令：

```bash
node -e "
const {logger} = require('./src/utils/logger');
logger.info('test', {
  password: 'secret',
  api_key: 'key',
  token: 'tok',
  nested: { secret: 'nested-secret' },
  array: [{ password: 'arr-pass' }]
});
" 2>&1 | grep -c "MASKED"
# 预期：5（全部掩码）或3（仅顶层），需根据实现判断合理性
```

4. DEBT-SEC-001诚实性（低风险）
审计重点：
- 是否明确说明"内存存储，重启清零"
- 是否说明"单机限流，不支持分布式"
- 清偿计划（Redis）是否 realistic
