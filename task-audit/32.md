SyncEngine 接口契约（来自 sync-engine.ts）：

```typescript
export interface ISyncEngine {
  sync(peerId: string): Promise<SyncResult>;
  push(peerId: string, chunkIds?: string[]): Promise<PushResult>;
  pull(peerId: string, chunkIds?: string[]): Promise<PullResult>;
  onConflict?: (local: Chunk, remote: Chunk) => Chunk;
}

export abstract class SyncEngine implements ISyncEngine {
  abstract readonly engineType: 'bidirectional' | 'unidirectional';
  protected offlineQueue: QueuedOperation[] = [];
  
  abstract sync(peerId: string): Promise<SyncResult>;
  abstract push(peerId: string, chunkIds?: string[]): Promise<PushResult>;
  abstract pull(peerId: string, chunkIds?: string[]): Promise<PullResult>;
}
```

双向同步核心实现（来自 bidirectional-sync.ts）：

```typescript
export class BidirectionalSync extends SyncEngine {
  readonly engineType = 'bidirectional' as const;
  
  async sync(peerId: string): Promise<SyncResult> {
    const connection = await this.dcManager.connect(peerId);
    try {
      const localManifest = await this.generateManifest();
      const remoteManifest = await connection.requestManifest();
      
      const diff = this.computeDiff(localManifest, remoteManifest);
      const conflicts = await this.resolveConflicts(diff.conflicts, connection);
      
      return this.executeSync(diff, conflicts, connection);
    } finally {
      connection.close();
    }
  }
  
  // offlineQueue 自动 flush 机制
  protected async flushQueue(): Promise<void> {
    while (this.offlineQueue.length > 0) {
      const op = this.offlineQueue.shift();
      if (op) await this.executeQueued(op);
    }
  }
}
```

Q1：offlineQueue 的内存持久化风险（DEBT-P2P-004）

- 现象：当前使用 `timestamp-based LWW`（Last Write Wins）
- 疑问：若设备时钟漂移（NTP 未同步），是否会导致错误覆盖？是否有 Vector Clock 版本？

⚠️ 特殊审计关注点（高风险）

1. P2P 连接的真实性与 Mock 风险
- 检查 `bidirectional-sync.ts` 是否直接实例化 `RTCPeerConnection`（真实）而非注入 `MockChannel`
- 检查 E2E 测试中 `fork` 的 `stdio` 配置是否为 `'ipc'`（必须 IPC 通信）
- 风险：若使用 `mock-socket` 或 `node-webrtc` 的 Mock 模式，视为测试虚假

2. 离线队列的内存泄漏风险（DEBT-P2P-004 深度）
- 检查 `offlineQueue` 是否有最大长度限制（如 `MAX_QUEUE_SIZE = 1000`）
- 检查失败操作是否无限重试（应有最大重试次数 `MAX_RETRIES = 3`）
- 检查 `flushQueue` 是否在 `sync` 前自动调用（避免队列无限堆积）

3. 冲突解决的时钟漂移风险（DEBT-P2P-001 深度）
- 检查 `onConflict` 是否仅比较 `timestamp`（危险）或包含 `logicalClock`（安全）
- 检查是否有 `Date.now()` 单调性校验（防止系统时间回拨）
- 建议：即使使用 LWW，也应记录 `conflictHistory` 供人工审计

4. 加密密钥的派生盐值风险（安全审计）
- 检查 `deriveKey` 是否接收 `peerId` 作为盐值（`salt: Buffer.from(peerId)`）
- 若无盐值，多设备场景可能产生相同密钥，违反前向保密

5. E2E 测试的竞态条件风险
- 检查 `laptop-b-sync.js` 是否有 `ready` 信号通知主进程（避免主进程过早开始测试）
- 检查 `fork` 后是否有 `await new Promise(resolve => laptopB.on('message', resolve))` 同步机制
- 检查测试结束是否 `laptopB.kill()`（避免僵尸进程）
