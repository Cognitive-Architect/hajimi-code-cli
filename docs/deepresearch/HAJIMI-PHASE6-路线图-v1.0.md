1. **5x 加速不可行**: 当前架构理论极限 3.8x，需架构重构才能突破 5x

3. **Phase6 推荐方向**: P2P 同步协议（功能扩展）+ 架构重构技术预研

**版本演进**: v3.0.0 (Phase5完成) → v3.1.0 (OBS修复) → v4.0.0 (Phase6)

## Phase6 路线选项

基于研究结论，Phase6 有三个可选方向，本路线图推荐**选项A为主，选项B为辅**的策略。

### 选项A: P2P 同步协议（推荐主路线）

**目标**: 实现去中心化的 WebRTC P2P 数据同步

**理由**:
- 5x 不可行，转向功能差异化
- P2P 是项目核心愿景（本地优先 + 去中心化）
- 技术债务已清偿，适合新功能开发

**技术架构**:

```
┌─────────────────────────────────────────────────────────────┐
│                      P2P 同步架构                            │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                  │
│  │  Node A  │──│  Node B  │──│  Node C  │  (WebRTC Mesh)   │
│  │ (本地)   │  │ (本地)   │  │ (本地)   │                  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘                  │
│       │             │             │                        │
│       └─────────────┴─────────────┘                        │
│              DataChannel (可靠传输)                         │
│       ┌─────────────┬─────────────┐                        │
│       ↓             ↓             ↓                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                  │
│  │ SyncMgr  │  │ Conflict │  │ Version  │                  │
│  │ (同步管理)│  │Resolver │  │ Vector   │  (CRDT)          │
│  └──────────┘  └──────────┘  └──────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

**Sprint 规划**:

| Sprint | 目标 | 工时 | 关键交付 |
|--------|------|------|----------|
| Sprint 3 | 信令服务器 + NAT穿透 | 2周 | STUN/TURN可用 |
| Sprint 4 | DataChannel可靠传输 | 2周 | 分片同步协议 |
| Sprint 5 | 冲突解决 + CRDT | 2周 | 最终一致性 |
| Sprint 6 | 加密 + 身份验证 | 2周 | E2E加密 |
| Sprint 7 | 性能优化 + 压力测试 | 2周 | 100节点测试 |

**版本**: v4.0.0-P2P  
**预计发布**: 2026-05-01  
**总工时**: 10-12 周

### 选项B: 架构重构（性能追求）

**目标**: WASM 主导架构，追求真 5x 加速

**理由**:
- 若有性能刚需（10万 QPS场景）
- OBS-001 预研积累的经验可复用

**技术架构**:

```
┌─────────────────────────────────────────────────────────────┐
│                   WASM 主导架构                              │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────────────┐   │
│  │                 WASM Runtime                         │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐          │   │
│  │  │ HNSWIndex│  │ ChunkMgr │  │ SyncMgr  │          │   │
│  │  │ (全内存) │  │ (WASM FS)│  │ (P2P)    │          │   │
│  │  └──────────┘  └──────────┘  └──────────┘          │   │
│  │                                                    │   │
│  │  ┌─────────────────────────────────────────────┐  │   │
│  │  │        Zero-Copy Memory Manager              │  │   │
│  │  │   (WASM Memory 直接暴露给 JS 视图)            │  │   │
│  │  └─────────────────────────────────────────────┘  │   │
│  └─────────────────────────────────────────────────────┘   │
│                         ↑                                   │
│                    最小化胶水层                              │
│                         ↓                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              JS Shell (轻量包装)                      │   │
│  │         init() | search() | sync()                  │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**Sprint 规划**:

| Sprint | 目标 | 工时 | 关键交付 |
|--------|------|------|----------|
| Sprint 3 | WASM内存管理器 | 2周 | 内存池实现 |
| Sprint 4 | 零拷贝API全量改造 | 3周 | search/insert零拷贝 |
| Sprint 5 | SIMD距离计算 | 2周 | 2-4x距离加速 |
| Sprint 6 | 兼容性层 | 2周 | 旧API兼容 |
| Sprint 7 | 性能验证 | 2周 | 5x达成验证 |

**版本**: v4.0.0-WASM  
**预计发布**: 2026-05-15  
**总工时**: 11-13 周  
**风险**: 高（架构重构，兼容性破坏）

### 选项C: IDE 自举开发环境

**目标**: 在 Hajimi 内部开发 Hajimi（自举）

**理由**:
- 证明系统成熟度
- dogfooding 发现隐藏问题

**功能清单**:
- Monaco Editor 集成
- 文件系统桥接（Hajimi存储 ↔ Editor）
- LSP 语言服务器（向量查询语法）
- 调试器（HNSW可视化）

**工时**: 10-12 周  
**推荐度**: 低（非核心功能）  
**建议**: 推迟至 v5.0.0

## 推荐路线：A+B 混合策略

### 核心策略

```
Phase6 (12周)
├── Sprint 3-4 (4周): 并行启动
│   ├── 主线: P2P信令 + NAT穿透 (选项A)
│   └── 支线: WASM内存管理器预研 (选项B技术储备)
│
├── Sprint 5-6 (4周): P2P核心
│   ├── DataChannel可靠传输
│   ├── 分片同步协议
│   └── 冲突解决 (CRDT)
│
└── Sprint 7-8 (4周): 收尾
    ├── P2P加密 + 身份验证
    ├── 性能优化 (可选接入WASM优化成果)
    └── 100节点压力测试
```

### 决策点

**Sprint 4 结束时的决策**:

| 条件 | 决策 |
|------|------|
| WASM内存管理器预研成功 + 性能提升显著 | 将选项B部分成果接入P2P |
| WASM预研不顺或收益有限 | 放弃选项B，全力投入P2P |
| P2P技术风险过高 | 回退到选项B（性能优化） |

### 高风险项
