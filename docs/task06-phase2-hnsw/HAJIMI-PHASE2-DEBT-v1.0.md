## 已知债务（预声明）

### DEBT-PHASE2-001: HNSW 库依赖风险

| 属性 | 值 |
|:---|:---|
| **描述** | 当前使用纯 JavaScript 自研 HNSW 实现，若未来需要更高性能，可能需要引入 `hnswlib-node` 等原生模块 |
| **优先级** | P1 |
| **影响** | 性能上限受限（纯 JS 约为 C++ 的 30-50%） |
| **缓解措施** | 1. 优化热点路径 2. 使用 TypedArray 3. 考虑 wasm 方案 |
| **计划解决** | Phase 3: 评估 wasm 绑定的可行性 |
| **状态** | 已接受 |

---

### DEBT-PHASE2-002: SimHash→Dense 向量编码损失

| 属性 | 值 |
|:---|:---|
| **描述** | SimHash-64 是稀疏二进制签名，转换为 128 维 dense 向量存在信息损失 |
| **优先级** | P1 |
| **影响** | 汉明距离与 L2 距离不完全等价，可能影响召回率 |
| **缓解措施** | 1. Hadamard 变换保持正交性 2. 保留 LSH 作为 fallback 3. 混合打分策略 |
| **计划解决** | Phase 2.1: 实现汉明距离直接支持 |
| **状态** | 已缓解 |

---

### DEBT-PHASE2-003: Termux 内存硬限制

| 属性 | 值 |
|:---|:---|
| **描述** | Termux 典型可用内存 < 500MB，严重限制索引规模 |
| **优先级** | P0-if-exceeded |
| **影响** | 单分片最大约 150K-200K 向量 |
| **缓解措施** | 1. 分片存储（16分片×100K=1.6M） 2. Lazy Loading 3. 内存压缩 4. 磁盘溢出 |
| **计划解决** | Phase 3: 实现磁盘溢出到 SQLite |
| **状态** | 已缓解 |

---

### DEBT-PHASE2-004: HNSW 索引构建耗时

| 属性 | 值 |
|:---|:---|
| **描述** | 100K 向量构建需 20-30s，阻塞主线程 |
| **优先级** | P2 |
| **影响** | 大批量导入时用户体验下降 |
| **缓解措施** | 1. Worker Thread（Node.js） 2. 分批构建 3. 后台构建 |
| **计划解决** | Phase 2.1: 实现异步构建 |
| **状态** | 已接受 |

## 隐藏债务（执行中发现）

### DEBT-PHASE2-005: JSON 序列化性能瓶颈

| 属性 | 值 |
|:---|:---|
| **描述** | HNSW 索引保存时使用 JSON.stringify，100K 向量序列化耗时 2-3s |
| **发现者** | 基准测试 |
| **优先级** | P2 |
| **影响** | 保存大索引时有明显卡顿 |
| **缓解措施** | 1. 增量保存（WAL） 2. 流式写入 3. 考虑二进制格式 |
| **计划解决** | Phase 2.1: 实现二进制序列化 |
| **状态** | 新发现 |

---

### DEBT-PHASE2-006: WAL 文件膨胀

| 属性 | 值 |
|:---|:---|
| **描述** | 频繁更新时 WAL 文件可能膨胀，未及时 checkpoint |
| **发现者** | 代码审查 |
| **优先级** | P2 |
| **影响** | 磁盘空间浪费，恢复时间增加 |
| **缓解措施** | 1. 定期 checkpoint 2. WAL 大小阈值自动触发 3. 压缩 |
| **计划解决** | Phase 2.1: 实现自动 checkpoint |
| **状态** | 新发现 |

---

### DEBT-PHASE2-007: 多线程安全风险

| 属性 | 值 |
|:---|:---|
| **描述** | HNSW 图结构非线程安全，并发修改可能导致不一致 |
| **发现者** | 架构设计 |
| **优先级** | P1 |
| **影响** | 并发写入可能损坏索引 |
| **缓解措施** | 1. 单线程写入 2. 读写锁 3. 队列化写入请求 |
| **计划解决** | Phase 2.1: 实现写入队列 |
| **状态** | 新发现 |
