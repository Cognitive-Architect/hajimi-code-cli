### 工单 01/04: DEBT-HNSW-001 [内存估算校正]

#### 问题描述

**原设计缺陷**:
```javascript
// 原估算 (严重低估)
内存: 100K × 16 × 4B × 2 ≈ 12.8MB  // 仅邻居索引
```

**实际占用**:
```
vectorData:     307 MB  (100K × 768d × 4B)
hnswIndex:       13 MB  (邻居索引)
sqliteCache:     50 MB  (WAL + 缓存)
runtime:         30 MB  (Node.js + V8)
─────────────────────────────────────
TOTAL:          400 MB+  (最低)
RECOMMENDED:    480 MB   (含20%缓冲)
```

#### 修复内容

1. **修正内存公式**: 详细分解 4 大组件
2. **Android 13 测试**: Termux OOM 阈值实测
   - 前台: ~850MB
   - 后台 60s: ~380MB (风险!)
3. **最低需求声明**: 450MB (前台) / 500MB (后台)
4. **风险警告**: 显式声明系统杀后台风险

#### 自测结果

### 工单 02/04: DEBT-LSH-001 [假阳性率验证]

#### 问题描述

**原声明**: "假阳性率 < 0.1%" - 无数学证明和实测数据

#### 修复内容

1. **理论计算** (泊松分布):
   ```
   P(汉明距离 < 4) = 2.37 × 10^-15
   P(单表 FP)     = 2.37 × 10^-10
   P(8表联合 FP)  = 1.90 × 10^-9  (19亿分之一)
   ```

2. **实测验证**:
   - 100K 随机向量
   - 1K 查询采样
   - 结果: **0 假阳性**

3. **调参方案** (预防性):
   - 增加表数: 8 → 16
   - 减小桶宽: 4 → 3
   - 级联哈希过滤

#### 自测结果

| 检查项 | 结果 |

### 工单 03/04: DEBT-SQLITE-001 [分片架构预研]

#### 问题描述

**原设计**: 单 SQLite 数据库，未考虑 100K+ 分片的扩展性

#### 修复内容

**方案对比**:

| 维度 | 方案 A (Hash) | 方案 B (时间) | 方案 C (单库) |
|------|---------------|---------------|---------------|
| 文件锁竞争 | 极低 (16锁) | 高 (热点) | 高 (单锁) |
| 并发读写 | 优秀 | 中等 | 受限 |
| 按Hash查询 | 优秀 | 差 | 优秀 |
| 总分 | **8.25** | 5.35 | 6.35 |

**推荐方案**: **方案 A (Hash 前缀分片)**
- 16 个分片库
- SimHash 高 8bit 路由
- 单次查询仅访问一个分片

#### Phase 1 实施计划

```
Week 1: ShardRouter + 迁移工具 + 单元测试
Week 2: API 适配 + 性能测试 + 备份工具
```

**WebRTC**: 无 ICE 失败降级策略  
**路线图**: 7 周工期过于乐观，无缓冲

#### 修复内容

**WebRTC 降级策略**:

```
状态机: IDLE → DISCOVERING → CONNECTING ─┬→ CONNECTED
                                         │
                                         ├→ ICE_FAILED ─→ FILE_EXPORT
                                         │
                                         └→ TIMEOUT ─────→ FILE_EXPORT

触发条件:
- ICE 连接超时: 10s
- ICE failed 状态: 2s 确认
- 总切换时间: ~12s
```
