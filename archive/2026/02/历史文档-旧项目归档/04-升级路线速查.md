# UPGRADE-ROADMAP-001 升级路线速查

**项目**: HAJIMI-MATH-001-CASCADE  
**目标**: MD5 → SHA256 → BLAKE3 平滑升级

---

## 🗺️ 三阶段总览

```
现在 (v3.0)      v3.1           v3.5           v4.0
  │              │              │              │
  ▼              ▼              ▼              ▼
┌─────┐      ┌─────┐      ┌─────┐      ┌─────┐
│MD5  │ ───→ │SHA  │ ───→ │BLAKE│ ───→ │BLAKE│
│0x02 │      │0x03 │      │0x04 │      │0x04 │(默认)
│默认  │      │默认  │      │默认  │      │     │
└─────┘      └─────┘      └─────┘      └─────┘
  │            │            │            │
  └────────────┴────────────┴────────────┘
              读兼容: 所有版本可读
              写可控: 默认新格式，允许兼容模式
```

---

## 📍 阶段 1 (v3.1): SHA256-256

### 格式变更
- **hash_type**: 0x03
- **条目大小**: 48B (原 32B)
- **结构**: simhash(8B) + sha256(32B) + length(4B) + seed(4B)

### 读写策略
| 操作 | 行为 |
|:---|:---|
| 读 | 支持 0x01/0x02/0x03 |
| 写 (默认) | 0x03 (SHA256) |
| 写 (兼容) | 可配置 0x02 (回滚用) |

### 迁移工具
```bash
hctx-migrate-v2-to-v3.js
# 输入: v2 (0x02) 文件
# 输出: v3 (0x03) 文件
# 需要: 原始块数据输入流
```

### 门禁
- 端到端耗时 ≤ 旧版 × 1.2 (沿用 <20% 预算)

### 债务
- **DEBT-SHA256-001**: SHA256 计算开销 (压测通过后关闭)

---

## 📍 阶段 2 (v3.5): BLAKE3-256 高性能

### 格式变更
- **hash_type**: 0x04
- **条目大小**: 48B (同 0x03)
- **结构**: simhash(8B) + blake3(32B) + length(4B) + seed(4B)

### 性能目标
- **单机**: > 1GB/s
- **实现路径**: Rust N-API 扩展 (优先) / WASM (备选)
- **门禁**: v3.5 耗时 ≤ v3.1 × 0.85

### 债务
- **DEBT-BLAKE3-001**: 原生/wasm 组件构建复杂度

---

## 📍 阶段 3 (v4.0): MD5 弃用

### Breaking Changes
| 操作 | v4.0 行为 |
|:---|:---|
| 写 | 只允许 0x04 (默认) / 0x03 (兼容窗口) |
| 读 0x02 | 只读支持，输出"需迁移"提示 |
| 读 0x03/0x04 | 正常支持 |

### API 清理
- 移除 MD5 强绑定字段
- 移除 legacy 全局 seed 配置

### 迁移工具
```bash
hctx-convert-legacy --from=0x02 --to=0x04
# 功能:
#   - 批量目录扫描
#   - 幂等执行
#   - 报表输出 (迁移率、失败列表、资源峰值)
```

### 债务清偿
- ✅ DEBT-RISK-006 (MD5 残余面)
- ✅ DEBT-SHA256-001 (若 BLAKE3 达标)

---

## 🔀 跨版本互操作矩阵

| 写版本 | 读版本 | 结果 | 策略 |
|:---|:---|:---|:---|
| v3.1 (0x03) | v3.1/v3.5/v4.0 | ✅ OK | 默认路径 |
| v3.5 (0x04) | v3.1 | ❌ 不支持 | fail-fast / 要求升级 |
| v3.5 (0x04) | v3.5/v4.0 | ✅ OK | 默认路径 |
| v4.0 (0x04) | v3.5 | ✅ OK | 渐进升级 |
| v4.0 (0x04) | v3.1 | ❌ 不支持 | 强制升级 |

⚠️ **关键原则**: 发布顺序必须是 **先升级读者，再升级写者**

## 📊 hash_type 分配表

| 值 | 哈希类型 | 条目大小 | 阶段 | 读写支持 |
|:---|:---|:---|:---|:---|
| 0x01 | Legacy SimHash-64 | 8B | v1/v2 | 只读 |
| 0x02 | SimHash-64 + MD5-128 | 32B | v3.0 | 读写 (v3.0-3.1) / 只读 (v4.0+) |
| 0x03 | SimHash-64 + SHA256-256 | 48B | v3.1 | 读写 |
| 0x04 | SimHash-64 + BLAKE3-256 | 48B | v3.5+ | 读写 (默认) |

---

## 🎯 格式方案选择

### 方案 A (固定大小) - 推荐
```
0x02: 32B
0x03: 48B  (+ sha256 32B)
0x04: 48B  (同 0x03)
```
✅ 简单，适合短期退役 MD5

### 方案 B (TLV 可扩展)
```
entry header: 16B 固定
payload: TLV 格式
  T=simhash64, L=8
  T=md5/sha256/blake3, L=16/32/32
```
✅ 未来扩展性好，❌ 解析成本高

- [ ] 破坏性输入可定位报错

### 性能门禁
- [ ] v3.1: SHA256 版 ≤ 旧版 × 1.2
- [ ] v3.5: BLAKE3 版 ≤ v3.1 × 0.85

### 完整性门禁
- [ ] 同一 chunk 在不同 hash_type 下判定一致
