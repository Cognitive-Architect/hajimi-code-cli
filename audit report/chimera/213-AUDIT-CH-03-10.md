### Q2：`add_turn` 方法的纯数据结构污染？

**审计结论**: ⚠️ **轻微污染，可接受，建议CH-04优化**

**当前实现**：
```rust
pub fn add_turn(&mut self, role: Role, content: String) {
    let timestamp = now_ms();  // ← SystemTime::now() 副作用
    // ...
}

fn now_ms() -> u64 {
    std::time::SystemTime::now()  // 系统调用
        .duration_since(std::time::UNIX_EPOCH)
        .unwrap_or_default()
        .as_millis() as u64
}
```

**污染评估**：
- **副作用范围**: 仅系统时间查询，无IO/网络/文件操作
- **可测试性**: 可通过`#[cfg(test)]`mock now_ms，或CH-04提取Clock trait
- **架构影响**: 轻微，不阻塞CH-04事件循环剥离

**对比方案**：
| 方案 | 行数 | 可测试性 | 复杂度 |
|:---|:---:|:---:|:---:|
| 当前(now_ms封装) | 56行 | 中(mock函数) | 低 |
| 纯数据(时间外置) | 50行 | 高 | 低 |
| Clock trait注入 | 60行 | 高 | 中 |

**判决**: 当前设计是**务实的中间状态**，不构成严重污染。建议CH-04提取`Clock` trait实现可测试性：
```rust
pub trait Clock { fn now_ms(&self) -> u64; }
impl ReplState<C: Clock> { pub fn add_turn(&mut self, ..., clock: &C) }
```
