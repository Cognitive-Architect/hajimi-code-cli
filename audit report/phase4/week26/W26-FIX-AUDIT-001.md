## 关键疑问回答（Q1-Q3）

### Q1：三栏申报标准是否真正解决"生产vs总计"口径混淆？
**审计结论**: ✅ **彻底解决，强制约束力明确**

**三栏强制机制**：
- 文档第11行明确声明："所有源文件行数申报必须提供三栏数据，禁止单栏'总计'糊弄"
- 第2.1节提供标准三栏申报表模板（文件路径/生产/测试/文档/总计/状态）
- 第2.2节要求每个工单收卷必须包含验证块（可复制执行的命令+输出）
- 第10节要求"未来所有ID-59派单必须引用本标准"

**防混淆设计**：
- 生产代码：明确排除`#[cfg(test)]`模块
- 测试代码：明确包含`#[cfg(test)]`全部内容
- 文档注释：单独统计，不计入生产或测试
- 总计：`wc -l`物理行数

### Q2：计算命令在复杂文件（如含多模块test）下是否精确？
**审计结论**: ✅ **验证通过，但需说明边界假设**

**V1-V3验证结果**（原始session.rs）：
| 验证项 | 命令/方法 | 输出 | 申报值 | 偏差 |
|:---|:---|:---:|:---:|:---:|
| V1 生产代码 | `#[cfg(test)]`从第108行开始 | **107** | 107 | 0 ✅ |
| V2 测试代码 | 195-107 | **88** | 88 | 0 ✅ |
| V3 总计 | `wc -l` | **195** | 195 | 0 ✅ |

**命令鲁棒性分析**：
- `sed '/^#\[cfg(test)\]/,/^}$/d'`假设：test模块从`#[cfg(test)]`开始，到第一个`}`结束
- session.rs结构：test模块确实从L108开始，到文件结束（L195），单一层级
- **边界情况**：嵌套`mod tests`、多个test模块、多行`#[cfg(test)]`属性可能需要调整

**建议增强**（可选Week 27补充）：
```bash
# 更鲁棒的方法：基于行号（已知test模块起始行）
head -n 107 src/memory/session.rs | wc -l  # 生产代码

# 或明确标记test模块边界
# #[cfg(test)]
// TEST-MODULE-START
...
// TEST-MODULE-END
```
