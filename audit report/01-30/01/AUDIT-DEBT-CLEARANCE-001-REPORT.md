### 系统性风险概述

不修复以下问题会导致：
1. **原型与生产落差**: LSH 验证使用简化 SimHash，生产环境可能表现不一致，导致假阳性率失控
2. **回归测试缺失**: 60项自测无统一运行脚本，后续迭代可能破坏已验证假设
3. **降级策略悬空**: WebRTC 降级停留在设计文档，缺少代码实现，故障时用户无退路

## 2. 风险详情（按严重性排序）

### R-001: LSH 验证与生产实现不一致（基线项）

**风险ID**: R-001  
**对应债务**: DEBT-LSH-001

- **风险描述**: `lsh-collision-sim.js` 使用简化版 SimHash 实现（随机投影），而生产环境声明使用 SimHash-64。不修复会导致：验证数据与生产行为脱节，假阳性率可能在真实场景超标
- **影响范围**: `src/test/lsh-collision-sim.js` 第 35-61 行，影响向量检索准确性
- **落地路径**:
  - **推荐方案**: 统一使用生产级 SimHash-64 实现替换简化版；涉及 1 文件，预计 2-4 小时
  - **替代方案**: 在报告中显式声明"简化实现，生产环境需复测验证"
  - **实施成本**: 2-4 小时 / 1 文件
  - **预期收益**: 消除验证与生产的实现差异，降低假阳性率失控风险
- **即时可验证方法**: 
  ```bash
  # 1. 检查生产 SimHash 实现路径
  find src -name "*simhash*" -o -name "*hash*" | head -20
  
  # 2. 对比两个实现的汉明距离分布
  node src/test/lsh-collision-sim.js --vectors 1000 --queries 100 --verbose
  # 预期: 汉明距离分布近似正态，峰值在 32 附近
  ```
- **基线判定**: 【必须修复】（影响正确性验证有效性）

  - **实施成本**: 2-3 小时 / 1 文件

  - **实施成本**: 4-6 小时 / 1 文件

### R-004: HNSW 内存估算缺少运行时监控代码（可选优化）

**风险ID**: R-004  
**对应债务**: DEBT-HNSW-001

- **风险描述**: 内存估算公式已修正，但无运行时监控代码。不修复会导致：用户设备接近内存极限时无预警，可能触发 OOM
- **影响范围**: `docs/DEBT-HNSW-001-FIX.md` 第 2 节，影响运行时稳定性
- **落地路径**:
  - **推荐方案**: 在存储初始化时添加 `process.memoryUsage()` 检查，低于 450MB 时发出警告；涉及 1-2 文件，预计 2-3 小时
  - **替代方案**: 在文档中增加"用户自检"章节，指导用户手动检查
  - **实施成本**: 2-3 小时 / 1-2 文件
  - **预期收益**: 提前预警内存不足，避免运行时被系统终止
- **即时可验证方法**:
  ```javascript
  // 验证代码片段
  const memUsage = process.memoryUsage();
  const freeMem = require('os').freemem();
  console.log('堆使用:', (memUsage.heapUsed / 1024 / 1024).toFixed(2), 'MB');
  console.log('系统空闲:', (freeMem / 1024 / 1024).toFixed(2), 'MB');
  // 预期: 能正确输出内存使用情况
  ```
- **基线判定**: 【可选优化】（建议 Phase 1 实施，可带债上线）
